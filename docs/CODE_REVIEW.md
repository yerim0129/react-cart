## 🔴 Priority 1 — 버그 / 즉시 수정

- [x] **로그인 후 이동 경로가 잘못됨**
  - 파일: `src/hooks/useAuth.ts:16`
  - **문제:**
    로그인이 필요한 페이지(예: `/cart`)에 비로그인 상태로 접근하면,
    앱이 "어디서 왔는지(from)"를 기억해두고 로그인 페이지로 보냅니다.
    그런데 로그인 성공 후 이동할 때 `navigate(-1)`을 쓰고 있는데,
    이건 "브라우저 뒤로가기"와 동일합니다.
    처음 앱을 열자마자 로그인 페이지로 들어온 경우처럼 히스토리가 없으면,
    앱 밖(이전 탭)으로 나가거나 로그인 페이지로 다시 돌아오는 루프가 생깁니다.
  - **수정 방향:**
    뒤로가기 대신, 기억해 둔 `from` 경로로 이동하도록 변경.
    `from`이 없으면 홈(`/`)으로 이동.

- [x] **찜하기 페이지에 쓰지 않는 코드가 남아있음**
  - 파일: `src/pages/WishlistPage.tsx:2`
  - **문제:**
    `useWishlistItems`라는 훅을 import 했지만 실제로 페이지 안에서 호출하지 않습니다.
    즉, 가져다 놓고 쓰지 않는 코드입니다.
    실제 찜 목록 필터링은 다른 방법(store의 `productIds`)으로 이미 처리 중입니다.
  - **수정 방향:**
    사용하지 않는 import 한 줄 삭제.

- [x] **로그인 / 회원가입 실패 메시지가 빈칸이 될 수 있음**
  - 파일: `src/pages/LoginPage.tsx:32`, `src/pages/RegisterPage.tsx` (동일 패턴)
  - **문제:**
    API 요청이 실패하면 에러 메시지를 화면에 보여주는데,
    지금은 에러 객체를 `as Error`로 강제 변환해서 `.message`를 꺼냅니다.
    그런데 에러가 항상 `Error` 객체 형식이 아닐 수 있습니다.
    예를 들어 네트워크 에러나 서버 에러는 다른 형식일 수 있고,
    그럼 `.message`가 `undefined`가 되어 에러 메시지가 비어 보입니다.
  - **수정 방향:**
    `error instanceof Error` 로 타입을 먼저 확인한 후 `.message`를 사용.
    아니면 기본 메시지(`'로그인에 실패했습니다.'`)를 보여주도록 처리.

## 🟠 Priority 2 — 설계 결함 / 잠재적 버그

- [x] **로그인 상태를 나타내는 값이 두 개라서 불일치 위험이 있음**
  - 파일: `src/store/authStore.ts`
  - **문제:**
    `authStore`에 `user`와 `isLoggedIn` 두 값이 따로 저장되어 있습니다.
    이 둘은 항상 같이 바뀌어야 하는데, 만약 어느 하나만 바뀌는 상황이 생기면
    "user는 있는데 isLoggedIn은 false" 같은 모순이 발생합니다.
    로그인 여부는 사실 `user`가 있는지 없는지만 보면 충분히 알 수 있습니다.
  - **수정 방향:**
    `isLoggedIn` 필드를 스토어에서 제거.
    사용하는 곳에서 `user !== null`로 로그인 여부를 판단하도록 교체.

- [x] **에러 화면이 React Query 기능을 쓸 수 없는 위치에 있음**
  - 파일: `src/main.tsx:19`
  - **문제:**
    현재 구조는 `ErrorBoundary > QueryClientProvider > App` 순서입니다.
    에러가 발생해서 ErrorBoundary가 활성화되면, QueryClientProvider 바깥에 있기 때문에
    에러 화면 안에서 React Query를 사용할 수 없습니다.
    예를 들어 에러 화면에서 "재시도" 버튼을 눌러 데이터를 다시 가져오는 기능을 만들 수 없습니다.
  - **수정 방향:**
    `ErrorBoundary`를 `QueryClientProvider` 안쪽으로 이동.
    `QueryClientProvider > ErrorBoundary > App` 순서로 변경.

- [x] **찜하기 토글 훅에서 서버 데이터를 중복으로 요청함**
  - 파일: `src/hooks/useWishlist.ts:18-21`
  - **문제:**
    `useWishlistItems` 훅과 `useWishlistToggle` 훅 모두 `['wishlists']` 쿼리를 각각 선언하고 있습니다.
    TanStack Query가 같은 키면 캐시를 공유해서 실제 API 요청은 1번만 일어나지만,
    코드가 같은 일을 두 곳에서 반복해서 선언하는 구조라 유지보수가 불편합니다.
  - **수정 방향:**
    `useWishlistToggle` 안에서 `useWishlistItems()`를 호출해서 데이터를 가져오도록 통일.

## 🟡 Priority 3 — 코드 품질 / 일관성

- [x] **날짜 포맷 함수가 두 곳에 따로 존재함**
  - 파일: `src/pages/admin/AdminOrdersPage.tsx:7`
  - **문제:**
    `src/utils/formatDate.ts`에 날짜를 포맷하는 함수가 이미 있는데,
    `AdminOrdersPage.tsx` 파일 안에 같은 역할을 하는 함수를 별도로 만들었습니다.
    두 함수의 출력 형식도 서로 달라서 날짜 표시가 페이지마다 다르게 보입니다.
  - **수정 방향:**
    `AdminOrdersPage` 안의 로컬 `formatDate` 함수를 지우고,
    `utils/formatDate`를 import해서 사용. 포맷도 통일.

- [x] **장바구니 데이터 마이그레이션에서 타입 처리가 불안전함**
  - 파일: `src/store/cartStore.ts:89`
  - **문제:**
    앱이 시작될 때 localStorage에 저장된 장바구니 데이터를 불러와서 정리하는 코드가 있습니다.
    이때 불러온 데이터를 `as CartState`로 강제 변환하는데,
    저장된 데이터의 구조가 현재 코드와 다를 경우(= 마이그레이션이 필요한 상황) 오히려 오류가 날 수 있습니다.
    마이그레이션의 목적 자체가 "예전 구조를 새 구조로 바꾸는 것"인데, 이미 새 구조라고 가정하고 처리하는 셈입니다.
  - **수정 방향:**
    `persistedState`를 더 느슨한 타입(`{ itemsByUser?: ... }`)으로 받아서
    optional chaining(`?.`)으로 안전하게 접근.

- [x] **상품 필터 옵션 객체가 매 렌더마다 새로 만들어짐**
  - 파일: `src/hooks/useProducts.ts:13`
  - **문제:**
    `useProducts({ category, sort, search })` 처럼 호출할 때,
    중괄호 `{}`로 묶인 옵션 객체는 렌더링될 때마다 새로 만들어집니다.
    TanStack Query는 쿼리키를 비교해서 "데이터를 새로 가져올지" 판단하는데,
    v5에서는 내용이 같으면 같다고 보기 때문에 현재는 문제가 없습니다.
    하지만 예상치 못한 불필요한 재요청의 원인이 될 수 있어서 인지하고 있을 필요가 있습니다.
  - **수정 방향:**
    옵션 객체 대신 `category`, `sort`, `search`를 쿼리키에 개별 값으로 분리하거나,
    호출 측에서 `useMemo`로 옵션 객체를 안정화.
